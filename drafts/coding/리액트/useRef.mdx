### useRef
`useRef`는 두 가지 기능을 한다.  
하나는 변수나 state처럼 값을 저장하는 역할, 다른 하나는 DOM(또는 컴퍼넌트) 셀렉터다.  

#### state, 변수와의 차이점
`useState`는 값이 바뀔 때마다 컴퍼넌트의 리렌더를 유발하고, 변수는 컴퍼넌트가 렌더링될 때마다 초기화된다.  
<br/>
그럼 <q>useRef</q>는?  
`useRef`는 값이 바뀌더라도 컴퍼넌트 전체의 렌더링을 야기하지 않는다.  
또한 컴퍼넌트가 리렌더링될 때도 기존의 값을 보전한다.  
<br/>
즉 뭔가를 안전하게 담아놓기 위해 타고난 보관함인 것.

### 사용방법
1. 값을 보관하는 경우  
초기값은 useState처럼 쓰면 된다. 제네릭에 타입을 생략해도 잘 유추해낸다.
```tsx index.tsx
const value = useRef<string>("")
```
<br/>
2. DOM 셀렉터의 경우  
초기값은 null로 명시하고, 요소의 ref 안에 걸어준다.
```tsx index.tsx
export default function App(){
  const inputRef = useRef<HTMLInputRef>(null)

  return (
    <>
      <input ref={inputRef} />
    </>
  )
}
```

#### 여기서 초기값이 `null`인 이유
최초 렌더링시 `useRef`가 선언되는 함수 구현부는 jsx 리턴부를 알기 전이다. 앞으로 어떤 DOM 요소가 그려질지, 어떤 요소를 참조할지 아직 알지 못하는 상태다.  
그렇기 때문에 DOM 셀렉터로서 `useRef`를 사용할 때는 초기값에 `null`을 준다. 아니면 줄 끊긴 우주비행사처럼 당황해 오류를 토해낸다.

DOM이 다 그려지는 시점에는 요소를 제대로 취득할 거라 예상되는 바, 이 시점은 `useEffect()`가 발동되는 시점이다. 이펙트 안에서 `.current`를 통해 요소가 제대로 걸렸는지 유효성을 한번 더 확인할 수 있다.

```tsx index.tsx
export default function App(){
  const inputRef = useRef<HTMLInputRef>(null)

  useEffect(()=> {
    inputRef.current && inputRef.current.focus()
  }, [])

  return (
    <>
      <input ref={inputRef} />
    </>
  )
}
```

### 각각의 타입
용도에 따라 타입도 두개로 양분된다.
```ts index.d.ts
  /** 값보관 useRef: 값 수정이 가능 */
  interface MutableRefObject<T> {
    current: T;
  }

  /** 
   * 셀렉터 useRef: 요소 변경이 불가능
   * 하지만 선택된 DOM 객체의 속성 수정은 가능하다.
   * @example
   * ref.current.value = "#$%"
   */
  interface RefObject<T> {
    readonly current: T | null;
  }
```

그러면 이제 아까 스크린샷의 에러 멘트가 이해된다.  
초기값 설정시 값을 비워두면, 이게 값 보관 `useRef`의 `undefined`로 읽히면서 타입이 `MutableRefObject`로 정해졌던 거다.  
그러니 `useRef`를 셀렉터로 쓸 땐 이제부터라도 꼭 `null`을 명시해야 하는 거셈. 뚜쉬 🐸
